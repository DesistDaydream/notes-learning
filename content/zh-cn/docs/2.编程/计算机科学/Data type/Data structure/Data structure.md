## Map vs 数组 取数速度对比

### 结论

**取数速度：Map 更快（在大多数场景下）**

|操作|数组|Map|
|---|---|---|
|按索引取数|O(1)|—|
|按 key 取数|O(n)|O(1)|
|按值查找|O(n)|O(n)|

### 细节说明

**数组按索引取数：O(1)**

go

```go
arr[0]  // 直接内存寻址，极快
```

如果你知道下标，数组是最快的，甚至比 Map 快，因为没有哈希计算开销。

**数组按 key 查找：O(n)**

go

```go
// 必须遍历
for _, v := range arr {
    if v.id == targetID { ... }
}
```

**Map 按 key 取数：O(1) 均摊**

go

```go
m[key]  // 哈希计算后直接定位
```

哈希冲突严重时最坏 O(n)，但实际中几乎不会发生。

---

### 怎么选？

- **已知下标** → 用数组，O(1) 且无哈希开销，内存局部性更好
- **按 key 查找** → 用 Map，O(1) vs 数组的 O(n)
- **顺序遍历** → 用数组，缓存友好，实际性能更好
- **频繁增删** → 用 Map，数组移动元素是 O(n)

---

### 一句话总结

> **按 key 查找：Map 赢；按下标访问：数组赢。** 大多数"取数"场景指的是 key 查找，所以通常说 Map 更快。

## 问题合理，但不够精确

"hash 还是线性表" 这个二选一本身没问题，但两者并不互斥——**hash 表的底层通常就是用数组（线性表）实现的**。

---

### 直接回答

|结构|底层实现|
|---|---|
|数组|线性表（连续内存）|
|Map|哈希表（底层仍是数组，加上哈希函数 + 冲突处理）|

### Map 的底层到底是什么？

Map 本质是 **哈希表**，而哈希表的核心是：

```
key → hash函数 → 索引 → 数组[索引]
```

所以 Map 的底层**还是数组**，只是多了一层哈希映射。

冲突处理方式一般两种：

- **链地址法**：数组每个槽挂一个链表（Go、Java HashMap）
- **开放寻址法**：冲突了就找下一个空槽（Python dict、Go map 部分实现）

---

### 问题哪里不够精确？

你问的是"hash **还是** 线性表"，暗示两者是对立的。  
实际上更准确的说法是：

> **数组是线性表；Map 是哈希表，哈希表是建立在线性表之上的高级结构。**

所以问"底层数据结构是什么"是合理的，但选项改成 **"线性表 还是 哈希表"** 会更准确。

## 跟线性表对应的概念

线性表的对立面是**非线性结构**。

数据结构按逻辑结构分为两大类：

```
数据结构
├── 线性结构        → 元素之间是一对一关系
│   ├── 数组
│   ├── 链表
│   ├── 栈
│   └── 队列
│
└── 非线性结构      → 元素之间是一对多 / 多对多关系
    ├── 树           （一对多）
    │   ├── 二叉树
    │   ├── B树
    │   └── 堆 ...
    └── 图           （多对多）
```

---

### 一句话总结

跟线性表对应的概念就是**非线性结构**，细分下来主要是**树**和**图**两大类。

哈希表比较特殊，它是一种**数据结构的实现方式**，底层用数组（线性），但从逻辑上看不属于上面任何一类，算是独立的一种——所以有时单独列出来，不强行归类。
